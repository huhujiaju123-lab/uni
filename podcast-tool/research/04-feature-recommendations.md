# podcast-viz 功能调整建议

> 基于用户需求研究的产品演进路线
> 日期：2026-03-07
> 当前版本：v2.4 | 18集内容 | 暂无自然流量

---

## 零、核心判断

### 产品定位校准

**当前定位**：播客内容的深度结构化分析工具
**建议定位**：**播客内容的「读图」体验** — 让一集播客在 30 秒内被理解、在 3 分钟内被消化、在 1 秒内被分享

**关键转变**：
- 从「全功能深度分析」→「分层体验」（预览层 / 消化层 / 深度层）
- 从「一刀切模板」→「类型感知」（知识型 vs 访谈型 vs 故事型的不同侧重）
- 从「个人学习工具」→「社交传播载体」（增加分享维度）

### 当前功能盘点

| 模块 | 状态 | 用户价值 |
|------|------|----------|
| 章节结构 + 导航 | ✅ 成熟 | 高 |
| 核心观点(key_points_grouped) | ✅ 成熟 | 高 |
| 金句收藏 | ✅ 有UI | 中（缺导出） |
| 思维导图 | ✅ 有 | 中高 |
| 内联图表(9种) | ✅ 有 | 高（**独有优势**） |
| 内容概览(content_overview) | ✅ 有 | 中 |

| 关键概念(key_concepts) | ✅ 有 | 中高 |
| 延伸阅读 | ✅ 有 | 低中 |

| 书单推荐 | ✅ 有 | 低 |

## 一、Phase 1 — 快速见效（1-2周）

### 1.1 新增：听前预览卡（Preview Card）

**问题**：用户打开页面直接面对深度分析，没有快速判断「值不值得看」的入口
**需求强度**：P0，覆盖所有播客类型

**建议方案**：
在 Hero 区下方、正文之前，插入一个「30秒预览」区块：

```
┌─────────────────────────────┐
│  📻 [播客名] · EP.XX        │
│                             │
│  「一句话总结」              │
│                             │
│  🏷️ #标签1  #标签2  #标签3  │
│                             │
│  👤 嘉宾: XXX（身份简介）    │
│  ⏱️ 时长: 1h23min           │
│  📅 发布: 2026-03-01        │
│                             │
│  适合你如果：                │
│  · 想了解 XXX 的底层逻辑     │
│  · 对 YYY 领域感兴趣         │
│  · 听过 ZZZ 想要进阶         │
│                             │
│  [▶ 继续阅读完整分析]       │
└─────────────────────────────┘
```

**实现路径**：
- analyzer.py 的 prompt 新增 `preview` 字段（one_sentence_summary 已有，补充 tags + target_audience）
- generator.py 新增预览区块渲染
- base.html.j2 在 hero 和 content 之间插入

**改动量**：小（prompt 加 3 个字段 + 模板加 1 个区块）

### 1.2 新增：社交分享海报

**问题**：用户想推荐播客给朋友时，只能发链接，缺少视觉传播物料
**需求强度**：P1，竞品蓝海

**建议方案**：
页面底部或浮动按钮「生成分享海报」，点击后弹出可保存的卡片：

```
┌─────────────────────┐
│  [播客封面/渐变背景]  │
│                     │
│  「最佳金句」        │
│                     │
│  ──────────────     │
│  📻 播客名 · EP.XX  │
│  一句话总结          │
│  #标签 #标签 #标签   │
│                     │
│  [二维码/链接]       │
│  ──────────────     │
│  podcast-viz 生成    │
└─────────────────────┘
```

**实现路径**：
- 方案A：纯 CSS/HTML 渲染 + html2canvas 截图（前端完成，无后端改动）
- 方案B：服务端 Playwright 截图（更稳定但更重）
- **推荐方案A**，轻量快速

**改动量**：中（JS 新增分享模块 + 海报模板 CSS）

### 1.3 增强：金句导出功能

**问题**：已有金句收藏 UI，但收藏后无法导出
**需求强度**：P1

**建议方案**：
收藏面板增加「复制全部」按钮，一键复制为 Markdown 格式：

```markdown
## 📻 [播客名] EP.XX — [标题]

> 金句1
> — 说话人

> 金句2
> — 说话人

来源：[链接]
```

**改动量**：极小（JS 加一个复制按钮 + 格式化函数）

## 二、Phase 2 — 类型自适应（2-4周）

### 2.1 播客类型识别 + 模板切换

**问题**：当前所有播客用同一套深度分析模板，对故事型/闲聊型过度分析
**需求强度**：P0（产品体验层面）

**建议方案**：

**Step 1：AI 自动识别播客类型**
在 analyzer.py 的 prompt 中新增类型判断：
```json
{
  "podcast_type": "knowledge|interview|narrative|news|casual",
  "podcast_type_reason": "一句话说明判断依据"
}
```

**Step 2：模板按类型调整渲染权重**

| 模块 | 知识型 | 访谈型 | 故事型 | 新闻型 | 闲聊型 |
|------|--------|--------|--------|--------|--------|
| 预览卡 | ✅ | ✅ | ✅(防剧透) | ✅ | ✅ |
| 章节结构 | 详细 | 详细 | 简要 | 详细 | 极简 |
| 核心观点 | **突出** | 突出 | 弱化 | 突出 | 隐藏 |
| 金句 | 正常 | **突出** | 正常 | 正常 | **突出** |
| 思维导图 | **突出** | 正常 | 隐藏 | 正常 | 隐藏 |
| 内联图表 | **突出** | 正常 | 隐藏 | 正常 | 隐藏 |
| 故事卡片 | 正常 | 正常 | **突出** | 弱化 | 正常 |
| 嘉宾信息 | 弱化 | **突出** | 弱化 | 正常 | 正常 |
| 自测 | 有 | 可选 | 隐藏 | 隐藏 | 隐藏 |
| 延伸阅读 | 有 | 有 | 有 | 有 | 隐藏 |
| 分享海报 | 有 | 有 | 有 | 有 | **突出** |

**实现方式**：不需要多套模板。在 generator.py 中根据 podcast_type 设置模块的 `visible` 和 `emphasis` 标志，Jinja2 模板用条件渲染。

**改动量**：中（prompt 加类型判断 + generator 加条件逻辑 + 模板加条件渲染）

### 2.2 访谈型增强：嘉宾信息卡 + 观点归属

**问题**：访谈类播客的核心价值是「谁说了什么」，当前观点未与说话人绑定

**建议方案**：
- participants 信息在页面顶部增加嘉宾卡片展示（头像/身份/代表作）
- arguments 和 key_points 绑定 speaker_id，渲染时标注发言人
- 可选：对话流视图（按时间线展示 A→B→A 的观点交替）

**改动量**：中

### 2.3 故事型适配：防剧透模式

**问题**：故事型播客的价值在悬念，可视化摘要等于剧透，会伤害体验

**建议方案**：
- 故事型播客的预览卡显示「本期讲述了一个关于 XXX 的故事」，不揭示结局
- 正文默认折叠，显示「已听完？展开完整回顾」
- 展开后以情节时间线为主（事件A → B → C），弱化观点提取
- 增加「情感标注」（💔 令人心碎 / 😮 意想不到 / 😊 温暖治愈）

**改动量**：中（模板加折叠逻辑 + prompt 调整故事型输出结构）

## 三、Phase 3 — 深度功能（4-8周）

### 3.1 Markdown/Notion 导出

**目标用户**：PKM 用户（5-10%，但付费意愿最高）

**建议方案**：
- 页面增加「导出」按钮，选择格式（Markdown / JSON / Notion API）
- Markdown 格式：标题 + 章节 + 要点 + 金句 + 概念
- 后期可接 Notion API 直接写入用户 workspace

**改动量**：中（前端生成 Markdown 字符串 + 下载）

### 3.2 原文时间戳跳转

**目标**：点击金句/章节 → 跳转到小宇宙对应时间点

**技术方案**：
- 小宇宙支持 deeplink：`https://www.xiaoyuzhou.com/episode/xxx?t=秒数`
- 当前已有 start_sec/end_sec，只需在渲染时生成对应链接
- 前提：需要知道该集在小宇宙的 episode_id（fetcher.py 已有）

**改动量**：小（模板加链接生成）

### 3.3 行动清单（Actionable Takeaways）

**目标用户**：知识型播客听众，想要「听完这集，我该做什么」

**建议方案**：
- prompt 新增 `action_items` 字段：3-5 条可执行的行动建议
- 渲染为 checkbox 列表，用户可勾选（本地存储）
- 只在知识型/商业型播客中显示

**改动量**：小（prompt 加字段 + 模板加渲染）

## 四、不建议做的事情

### 4.1 不要试图服务所有播客类型

闲聊型播客（日谈公园、凹凸电波）的可视化价值极低。与其勉强生成一个「闲聊要点总结」让用户觉得「这期好像没啥内容」，不如直接不支持或只生成极简的分享卡片。

**建议**：在首页或生成结果页，对闲聊型播客显示提示「本期播客以轻松对话为主，我们为您精选了最佳金句和分享卡片」，而不是强行输出完整分析。

### 4.2 不要正面对标 Snipd/Podwise

这两个产品在「个人学习笔记」赛道已经成熟，且有付费用户基础。podcast-viz 的差异化在于：
- **视觉化**（图表、思维导图、海报 — 竞品是纯文字列表）
- **社交传播**（分享卡片 — 竞品主攻个人消费）
- **互动体验**（自测、收藏 — 竞品是静态文档）

应该强化这三个差异点，而不是去做又一个 Markdown 笔记工具。

### 4.3 不要过早追求完美的说话人识别

当前 Deepgram 的 diarization 已经够用。WhisperX 说话人识别（PRECOMPUTE_STRATEGY Phase 4）成本高、准确率不稳定，且只有访谈型播客强需要。建议等用户量上来后再投入。

### 4.4 不要增加太多模块

当前已有 15+ 个内容模块（章节、观点、金句、故事、图表、概览、论证、概念、延伸、思维导图、自测、推荐……），信息密度已经很高。增加新模块前，先考虑：
- 能否合并到现有模块？
- 这个模块对主要用户类型是否真正有价值？
- 加了之后页面会不会太长？

**原则**：宁可做少做精，不要大而全。
