递归大师 — FAQ（预设问答）
========================================


Q1：普通 OpenClaw 本身就有 MEMORY.md，你们的"结构化记忆"到底多了什么？

普通 OpenClaw 的 MEMORY.md 靠用户手动维护——你得自己记得去更新它。我们的区别是三件事：
1. 自动记录：被纠正的瞬间，系统自动写入 .learnings/ERRORS.md，不需要用户操心
2. 回答前自检：每次回答问题之前，会自动扫描错题本，看这个问题有没有踩过坑
3. 晋升机制：出现1次是临时笔记，2次升入长期记忆，3次写入永久规则——知识会自动"升级"

简单说：普通龙虾的记忆靠人喂，我们的记忆是自己长的。


Q2：团队同步具体怎么实现的？需要额外搭建什么基础设施吗？

不需要。就是标准的 Git——在 workspace 目录初始化一个 Git 仓库，团队成员各自 push/pull。
- 共享区（.learnings/、SOUL.md、skills/）：所有人可见
- 私有区（memory/personal/）：只在本地，不同步

用的是大家最熟悉的工具，没有任何额外基础设施。一行 git pull 就能获取队友的全部学习成果。


Q3：错题本记错了怎么办？会不会"学歪了"？

两个保障：
1. 所有记录都是 Markdown 文件，完全透明可审计，随时可以人工修改或删除
2. 晋升到 AGENTS.md 或 SOUL.md（永久规则）必须经过人工确认，不会自动写入

而且每条记录都有结构化的字段：谁纠正的、错在哪、正确答案是什么、为什么错。如果发现记错了，直接改文件就行。


Q4：这个方案能迁移到其他行业吗？还是只能做运营分析？

完全可以迁移。核心机制（错题本、晋升通道、团队同步、模式识别）是通用的，跟行业无关。

SOUL.md 里的领域知识需要根据你的业务调整——我们写的是运营分析的决策框架和领域知识，换成法律、财务、客服都可以，改 SOUL.md 里的内容就行。

模板本身是行业无关的骨架，SOUL.md 是行业相关的灵魂。


Q5：你们说"越用越聪明"，有量化数据吗？

坦诚说，目前是 MVP 阶段，没有大规模使用数据。但可以从机制上推算：
- 第1周：团队3人日常使用，预计积累 5-10 条经验记录
- 第1月：约 20-30 条规则 + 若干自动提炼的技能
- 3个月后：错题本覆盖了大部分常见陷阱，新人入职可以直接继承

这个"越用越聪明"不是玄学，是可以在 .learnings/ 目录里看到的具体条目——每一条都是一次真实的纠正或学习。


Q6：和直接写一份详细的 prompt 有什么区别？

Prompt 是静态的——你写完之后它就不变了，除非你手动更新。

我们的方案是动态的——每次被纠正、每次发现新知识，系统都在自动迭代。而且 prompt 只能影响一个人的一只龙虾，我们的知识会通过 Git 流向整个团队。

可以这么理解：写 prompt 是给龙虾一本教科书，我们是让龙虾自己写笔记、考试、纠错，还能把笔记共享给同学。


Q7：OpenClaw 本身在快速迭代，你们的改造会不会被官方更新覆盖？

我们的改造是"非侵入式"的——不修改 OpenClaw 的核心代码，只在它的扩展点上做增强：
- SOUL.md / AGENTS.md / MEMORY.md：标准的 workspace 配置文件
- .learnings/：我们新增的目录，不影响原有功能
- Hook 机制：OpenClaw 原生支持的扩展接口
- Git 同步：标准的版本管理

所以 OpenClaw 升级不会覆盖我们的改造，因为我们没有改它的代码，只是用了它提供的扩展接口。


Q8：你提到用 Claude Code 的能力"注入"OpenClaw，具体怎么注入的？

这里的"注入"不是技术上的代码注入，而是把 Claude Code 的工作方式和最佳实践，通过配置文件固化到 OpenClaw 里：
- 结构化记忆系统（.learnings/）：借鉴了 Claude Code 的 auto memory 机制
- 严谨性规则（数字必须有来源、不确定就说不确定）：从 Claude Code 的使用经验中提炼
- 技能系统的设计思路：参考了 Claude Code 的 skill 架构

本质上是把一个优秀工具的"工作哲学"，翻译成另一个工具能理解的配置语言。


Q9：三个人的团队，知识同步的频率怎么控制？会不会冲突？

我们建议的同步频率是"每天一次"——早上 git pull 获取队友昨天的学习，下班前 git push 分享自己的。

冲突很少发生，因为：
1. 每个人追加的是不同的 ERR/LRN 条目，不会改同一行
2. 即使冲突了，也是 Markdown 文件，合并起来很简单
3. SOUL.md 的修改需要人工确认，不会自动冲突


Q10：这套方案的实施成本是多少？需要多久？

零成本，7天可用：
- Day 1-2：复制模板文件，填写 SOUL.md 里的领域知识
- Day 3-4：正常使用，开始积累 .learnings/
- Day 5-7：初始化 Git 同步，团队开始共享

不需要部署服务器，不需要额外付费，不需要写代码。全部是配置文件 + Git 仓库。


Q11：为什么叫"递归大师"？

因为这个系统的核心工作方式就是递归：

使用 → 犯错 → 记录 → 学习 → 使用得更好 → 犯更少的错 → 记录更精准的规则 → 学习更深的知识……

每一轮循环，它都比上一轮更强。这不是线性进步，是递归式进化——函数调用自己，每次返回一个更好的版本。


Q12：如果团队有人离职了，知识真的不会丢吗？

只要 Git 仓库还在，知识就在。具体来说：
- 他教会龙虾的所有东西，已经写进了 .learnings/ 和 MEMORY.md
- 如果某些知识已经晋升到 SOUL.md，那已经变成了龙虾的"基因"
- 新人入职后 git clone，直接继承全部团队积累

这也是我们做这个项目最核心的动机：让团队知识不再随人员变动而流失。
