# 数据分析底层哲学

> 来源：2026-03-12 提频任务方案复盘，用户确认"非常到位"。
> 适用于所有分析场景：AB实验、活动方案、券包设计、实验报告、月报周报。

## 一句话

**你不是在做分析，你是在用数据讲一个让人无法反驳的故事，故事的结尾是一个明确的行动。**

---

## 五层思考框架

### 第一层：先想清楚"要做什么决定"

不管什么场景，第一个问题永远是：**这份文档结束后，读的人要拍板什么事？**

| 场景 | 决策 |
|------|------|
| 提频任务 | 任务参数定多少？ |
| AB实验报告 | 全量推/迭代/停？ |
| 新客券包 | 券面多少？组合怎么配？ |
| 分享有礼 | 哪个环节值得加资源？ |
| 月报 | 下个月重点投什么？ |

从决策倒推需要哪些证据，而非从数据出发"看看能分析出什么"。

### 第二层：论证结构是"律师辩护"，不是"学术论文"

- 学术论文：我看了所有数据 → 这是发现 → 供参考
- 律师辩护：我的主张是 X → 证据 1/2/3 → 所以必须 X

每一条数据都在服务一个论点，没有"顺便看看"的数据。

### 第三层：信息筛选 — "删掉这个，结论会变吗？"

写完每张表/每个图，问自己：如果删掉这个，结论会变吗？
- 会变 → 保留
- 不会 → 删掉

分析过程中看过的数据可能有 20 张表，最终文档里只需要 5 张。

### 第四层：复杂度留在计算过程，不留在文档

- Poisson 模型、统计检验 → 计算工具，口头回答"怎么算的"
- 文档只呈现结论："预计转化率从 56% 提升至 66%，增量约 10,000 用户"
- 读者不需要知道过程，只需要相信结论合理

### 第五层：参数可信度 = 文档可信度

一个不靠谱的参数让整份文档信任归零。
- 不确定的参数，宁可空着问，不要自己编
- 用历史数据或保守估计，不乐观假设
- 券成本、参与率、转化率 → 必须对齐实际业务值
