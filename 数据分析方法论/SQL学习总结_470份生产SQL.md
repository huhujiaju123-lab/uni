## 二、三个核心认知

### 认知一：数据不是数字，是决策的语言

470 份 SQL 不是 470 个查询。它们是 Lucky US 过去一年多的决策轨迹。

我能从 SQL 的演进中看到业务的思考脉络：
- **早期 SQL**：简单的 `SELECT COUNT(*)`，关注「今天卖了多少杯」
- **中期 SQL**：出现 CTE 链、DATEDIFF 留存框架，关注「用户留下来了吗」
- **近期 SQL**：Rolling Cohort、RFM 分层、折扣深度交叉分析，关注「不同人群要怎么差异化运营」

这说明 Lucky US 的数据能力正在从 **描述性分析**（发生了什么）向 **诊断性分析**（为什么发生）和 **决策性分析**（该怎么做）跃迁。

一个更具体的例子：价格实验的 SQL 从最初只看「A 组 vs B 组的转化率」，到后来按 Recency × Frequency 交叉分层，再到按首单 AOV、折扣深度、杯数三维分层。这不只是 SQL 变复杂了，而是团队对「价格敏感度」的理解从一个数字变成了一张用户画像。

### 认知二：好的数据架构是「让正确的事容易做」

学了这么多 SQL 后，我发现一个强烈的模式：**90% 的日常分析只需要 3 张 ADS 表**。

```
ads_mg_sku_shop_sales_statistic_d_1d  → 经营看板的一切
ads_user_order_rep_info_d_nw          → 周复购的一切
dws_mg_log_user_screen_name_d_1d      → 漏斗转化的一切
```

这三张表的设计者做了一个正确的决定：把「分析师最常问的问题」预计算好。这让日报从「写 200 行 SQL 跑 5 分钟」变成「写 20 行 SQL 秒出」。

但同时我也看到了另一面：当分析需求超出预聚合范围（比如需要订单级明细、杯级单价、促销归因），分析师就不得不回到 ODS 层，面对 `t_order_item JOIN v_order` 的性能困境。

**这给我的启示是**：数据架构的核心不是存多少数据，而是让最常见的 80% 分析需求在 ADS 层「秒出」，同时为剩下 20% 的深度分析保留 ODS 层的灵活性。Lucky US 的数仓设计在这个平衡上做得不错。

### 认知三：SQL 模式的复用度极高，但需要显式沉淀

470 份 SQL 里，我发现了大量的重复模式。以下 6 个模式覆盖了 80% 的查询逻辑：

| # | 模式 | 出现频率 | 本质 |
|---|------|---------|------|
| 1 | 时区转换 + 租户过滤 + 测试店排除 | 几乎每一份 | 数据清洗起手式 |
| 2 | CTE 链式留存（target → future → rate） | 高频 | 用户行为追踪 |
| 3 | CASE WHEN 多维分层（RFM/AOV/折扣/杯数） | 高频 | 人群切片 |
| 4 | CROSS JOIN 日期序列 × 人群 | 中频 | 时间维度展开 |
| 5 | 窗口函数累计（UNBOUNDED PRECEDING） | 中频 | 趋势和环比 |
| 6 | 首单识别（MIN date GROUP BY user_no） | 高频 | 新客定义 |

**但问题是**：这些模式在 470 份 SQL 中是隐式存在的。每份新 SQL 都在重新手写这些模式，而不是从一个统一的模板库调用。

我做的知识库和 skill 文档更新，本质上就是把这些隐式模式**显式化**——让以后每次写 SQL 不需要从头发明轮子，而是从已验证的积木块组装。

## 三、几个具体发现

### 发现 1：「杯量」这个指标比看起来复杂

很多人直觉认为杯量 = 订单数。但在 Lucky US 的语境里：
- 杯量 = `COUNT(*)` from t_order_item（每行一杯）
- 一笔订单可能包含 2-3 杯（多杯用户）
- 套餐（spu_type = 2）的 sku_num 可能 > 1

这意味着「杯量」和「订单数」是两个不同的指标，它们的差值反映了**多杯消费的深度**。多杯率是一个被低估的增长杠杆。

### 发现 2：外卖渠道是一个独立世界

channel 8/9/10（DoorDash/Grubhub/UberEats）不只是一个渠道标记，它代表一个完全不同的用户行为模式：
- 不经过 APP 漏斗（没有 menu → productdetail 的路径）
- 客单价结构不同（平台加价）
- 用户画像不同（type = 5 的外卖用户需要剔除）

分析时如果不分离 Pickup 和 Delivery，很多指标会被稀释。

### 发现 3：时间窗口的选择是一个设计决策

同一个留存率指标，在不同报表里的定义可能不同：
- 日报用 D7 留存（首单后 7 天内再购）
- 周报用周复购（本周买且上周也买）
- 月报用月复购（本月买且上月也买）
- 实验用 D0/D3/D7/D14 四个窗口

这不是因为不统一，而是因为**不同的决策周期需要不同的时间尺度**。日报要快反应，月报看趋势，实验要可比性。理解这一点，就不会在被问「留存率是多少」时只给一个数字。

### 发现 4：实验分析的精髓是「控制混淆」

从 470 份 SQL 里提炼出的 3 种实验分析框架（Fixed Cohort / Rolling Cohort / Daily Dimension），它们的核心区别不是 SQL 写法，而是在回答「如何控制时间维度的混淆」：

- Fixed Cohort：锁定开始日人群，适合短期评估，但会包含「越晚进组效果越好」的时间偏差
- Rolling Cohort：每日新入组单独追踪，消除时间偏差，但计算量大
- Daily Dimension：只看当天，不累计，最简洁但丢失了纵向追踪能力

**真正的洞察是**：没有哪种框架是「最好的」。好的实验分析是用 3 种框架从不同角度看同一个问题，如果结论一致，信心就强；如果矛盾，就要追问为什么。

## 四、对未来工作的建议

### 短期（可立即执行）

1. **日报自动化已准备就绪**：5 个 SQL 模块已模板化，说「出日报」就能跑
2. **新实验可直接套用 3 种框架**：不需要每次重写分流、追踪、分层的 SQL
3. **分享有礼报告有完整模板**：六大板块 + 全链路漏斗

### 中期（需要讨论）

4. **建议把「试喝券」单独拆出来追踪**：promotion_name = 'Sampling Coupon' 是一个独立的获客渠道，目前混在新客指标里
5. **外卖渠道的用户生命周期需要独立分析**：Delivery 用户和 Pickup 用户的留存曲线可能完全不同
6. **券使用率的「同日核销率」是一个好指标**：它衡量的不是「最终有多少人用券」，而是「券发出去后多快被激活」——这是紧迫感的代理变量

### 长期（数据架构层面）

7. **缺一张 ADS 层的「用户日级宽表」**：目前用户级分析都要回 ODS 层算，如果有一张每日更新的用户宽表（包含 RFM 标签、留存状态、上次下单日期、累计杯量等），很多分析可以从「写 200 行 CTE」变成「SELECT WHERE」

## 五、写在最后

读完 470 份 SQL，我最大的感受是：**数据分析的核心竞争力不是写 SQL 的速度，而是对业务的理解深度**。

同样一个「新客7日留存率下降了 5%」的事实，初级分析师会报数，中级分析师会拆维度（哪个门店/渠道/时段下降了），高级分析师会问「下降的这 5% 是因为获客质量变了（来源渠道变化？折扣深度变化？）还是留存动作变了（推送频次？券策略？）」。

470 份 SQL 教会我的，不只是哪张表有什么字段。而是 Lucky US 的团队在用什么样的视角理解用户、理解增长、理解一杯茶背后的人的行为。
