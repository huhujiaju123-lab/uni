# 活动/任务方案设计原则

> 来源：2026-03-12 提频任务方案复盘，对比用户发给老板的 PDF 和我生成的版本。

## 核心思路：数据推导方案，而非方案找数据验证

### 原则 1：问题驱动
- 开头 3 个**具体业务问题**（不是"分析频次现状"这种模糊目标）
- 例："用户买多少杯后留存稳定？""5杯免1杯是否合理？"
- 结尾用问答表回扣这 3 个问题，形成闭环

### 原则 2：用户旅程框架，而非指标罗列
- 按用户生命阶段组织：首杯 → 复购 → 习惯 → 忠诚
- 每阶段回答：多少人？卡在哪？多久转化？转化率？
- **禁止**按指标类型罗列（频次分布 → 迁移矩阵 → 星期分布 → …）

### 原则 3：方案从数据自然推导，每个参数指向一条数据
- 任务周期 7 天 ← "2/3 转化发生在 7 天内"
- 目标 3 杯 ← "3 杯后转化率从 40% 跳升到 51%"
- 奖励阶梯 ← 各阶段流失率递减（46% → 39% → 26% → 19%）
- **禁止**"我觉得""建议"，每个参数后面必须跟数据出处

### 原则 4：测算用最简单直觉的模型
- 核心公式：`用户数 × 现状转化率 × 策略提升 = 增量`
- 一张表说清：阶段 / 队列人数 / 现状转化率 / 策略后预期 / 增量
- **禁止**用 Poisson 分布、"免费增量区"等需要解释的概念
- 老板看完 5 秒能理解

### 原则 5：参数必须对齐实际业务数值
- 券成本：问清实际面值，不自己推算（曾经用 $1.80 实际是 $1.30）
- 参与率：用历史数据或保守 5-10%，不乐观假设（曾经用 15%/25% 实际是 5%/10%）
- 杯均价：用实收均价
- **生成前先确认关键参数**，避免反复返工

### 原则 6：文档格式 — 表格主导，文字极少
- 表格 > 图表 > 文字
- 关键发现不超过 3 条，每条一句话
- 不做多余解释，数据自己说话
- 总结用问答表收尾（4 行以内）
- 参考模板：`~/Downloads/US用户运营任务方案.pdf`

## 常见错误清单
1. 先有结论再找数据验证（应该反过来）
2. 指标罗列而非用户旅程叙事
3. 模型过度复杂（Poisson/概率分布 → 老板看不懂）
4. 参数不对齐实际（券成本、参与率脱离业务）
5. 文字解释太多（应该让表格自己说话）
