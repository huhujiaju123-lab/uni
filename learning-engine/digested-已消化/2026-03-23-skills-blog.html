引言

Anthropic 的 Claude Code 团队发了一篇内部实战总结——他们内部活跃使用的 Skills 已经有几百个。文章里讲了 9 种 Skill 类型、8 个编写技巧、分发和组合策略。

问题是：如果你不是专业程序员，而是一个每天跑 SQL、看数据、做实验分析的人，这些东西对你意味着什么？

用数据分析师的语言，把文章里每一个概念翻译成你已经在做的事情。读完之后，你不只是"了解了 Skills"，而是能说出"我明天要做哪个 Skill"。

第一部分

Skill 到底是什么

原文怎么说的

文章开头就纠正了一个常见误解：

"我们经常听到一个误解，认为 Skills 只不过是 markdown 文件。但 Skills 最有意思的地方恰恰在于它们不只是文本文件——它们是文件夹。"

一个 Skill 文件夹可以包含：指令文档、参考代码、脚本、数据文件、配置……Claude 会在需要的时候主动去翻阅这些文件。

翻译成数据分析语言

你跑一个复杂分析——比如 AB 实验月报——不是一条 SQL 能搞定的。你需要：

  • 分流质量检查的 SQL 模板
  • 核心指标对比的 SQL 模板
  • 各种口径约定（杯量怎么算、实收只看饮品、外卖要单独拆……）
  • 踩过的坑（ODS 表会超时、tenant 必须过滤、试喝券要剔除……）
  • 最终报告的 HTML 模板

把这些全部放进一个文件夹，写一份说明告诉 Claude "当用户说做 AB 实验分析时，按照这个来"——这就是一个 Skill。

Skill = 你脑子里那套"做某件事的全部经验"，文件夹化之后交给 AI。

产品验证

原文

描述如何测试或验证代码是否正常工作。搭配 Playwright、tmux 等工具。值得安排一个工程师花一周时间打磨验证 Skills。

数据分析翻译

这就是数据质量校验。你跑完一个查询，怎么知道结果是对的？现在你靠肉眼看——数字对不对、量级合不合理、和上周比偏差大不大。这些检查完全可以自动化。

具体场景：你做 AB 实验分流质量检查。现在的流程是：

  1. 写 SQL 看各组人数
  2. 人肉判断"人数差异 < 5% 就算合格"
  3. 写 SQL 看各组基础特征
  4. 人肉判断"p 值 > 0.05 就算无显著差异"

Skill 化：把这四步变成一个验证脚本。Claude 跑完分析后自动执行，如果不合格就告警"分流质量存疑"。你不用记住该检查什么——Skill 替你记住。

你已有的 i

ab-experiment skill 里有分流质量检验的 SQL 框架。

下一步 +1

把"合格标准"写成可执行的断言：

验证规则：
- 各组人数偏差 < 5%
- 关键基础指标（客单价、到访率）p值 > 0.05
- 涨价组的活跃用户占比与对照组差异 < 3pp
- 如果不通过，输出："⚠️ 分流质量存疑，建议核查"

第三部分

八个编写技巧全解

文章说"确定了要做什么 Skill 之后，怎么写呢？"以下是他们总结的技巧。

1

不要说显而易见的事

Claude 对你的代码库已经很了解。Skill 应该把重点放在能打破 Claude 常规思维模式的信息上。

你不需要在 Skill 里写"SELECT 是查询语句"——Claude 知道。你要写的是 Claude 反复犯错的那些事。

比如 Claude 写 SQL 时有几个"默认坏习惯"：

  • 默认用 COUNT(DISTINCT order_id) 算杯量（错的，应该用 COUNT(*) from order_item）
  • 默认不加 tenant 过滤（漏了会查到其他租户的数据）
  • 默认用 create_time 而不是 local_dt（时区没转）

这些才是 Skill 应该写的——打破 Claude 的默认错误模式。

文章举了 frontend-design 这个 Skill 的例子：它专门避免 Claude 默认的"Inter 字体+紫色渐变"审美。翻译到你的场景：你的 Skill 专门避免 Claude 默认的"用错表+忘过滤+杯量算法错"。

2

建一个踩坑点章节

这是整篇文章最重要的一句话。

踩坑点不是你坐下来一次性写完的——是你每次用 Claude 跑数，发现它犯了一个错，就补一条。

举个真实例子。你在做 0311 价格实验分析时发现：涨价组2的分流质量存疑（到访率低 20pp），你当时调查后得出结论"组间构成不均衡，整体对比无效，必须分层分析"。这就是一个踩坑点：

踩坑点 #7：AB实验不能只看整体
发现日期：2026-03-11
场景：0311价格实验，涨价组2有25%活跃用户，其他组仅11%
教训：必须按用户生命周期分层（0-15天/16-30天/30+天），
     否则辛普森悖论会让整体对比完全失效

你积累 10 个这样的踩坑点，Claude 以后做实验分析时就会自动分层、自动检查构成均衡性。这比任何 prompt 工程都管用。

3

利用文件系统与渐进式披露

把整个文件系统当作上下文工程和渐进式披露的工具。告诉 Claude 你的 Skill 里有哪些文件，它会在合适的时候去读取。

不要把所有信息塞在一个 markdown 里——会撑爆上下文窗口。渐进式披露的意思是：只在需要的时候才加载详细信息。

你的 luckyus-data-query skill 有一个 650 行的 knowledge-base.md。每次 Claude 启动时都要读这 650 行。但其实 90% 的查询只用到其中 3 张表。

更好的做法：

skill.md（主文件，50行）
  → "门店杯量查 references/sales-table.md"
  → "用户留存查 references/retention-table.md"
  → "实验分析查 references/experiment-tables.md"
  → "踩坑点汇总见 gotchas.md"

Claude 看到主文件的目录，只在需要"实验分析"时才去读 experiment-tables.md。这就是渐进式披露——告诉 Claude 有什么，但不要一次全塞给它。

4

不要把 Claude 限制得太死

给 Claude 需要的信息，但留给它适应具体情况的灵活性。

你的 Skill 应该告诉 Claude "杯量必须用 COUNT(*) 从 order_item 表算"（这是硬规则，不能灵活），但不应该规定"输出格式必须是三列表格"（这应该根据具体分析灵活调整）。

区分两种指令：

  • 硬规则（不可违反）：数据口径、过滤条件、关联方式
  • 软指引（可适应）：输出格式、分析维度选择、可视化方式
5

考虑好初始设置

有些 Skills 需要用户提供上下文来完成初始设置。好的做法是把设置信息存在 config.json 里。

你的 AB 实验 Skill 需要知道"当前在跑哪些实验"。这个信息每隔几周就变。可以做一个 config.json：

{
  "active_experiments": [
    {"name": "0311价格实验", "start_date": "2026-03-11", "groups": 3},
    {"name": "0212老客涨价", "start_date": "2026-02-12", "groups": 2}
  ],
  "default_metrics": ["杯量", "实收", "访购率", "ITT人均饮品实收"]
}

Claude 读到这个配置，就知道当前有什么实验在跑，不用你每次重复说明。

6

description 字段是给模型看的

description 不是摘要——它描述的是何时该触发这个 Skill。好的 description 读起来更像 if-then 条件。

这个很多人写错。对比：

写成功能说明写成触发条件
"AB 实验分析技能，支持分流检验和效果分析" "当用户提到 AB 实验、分流、实验效果、置信度检验时使用"
"Lucky US 数据查询工具" "当用户需要跑数、写 SQL、分析业务数据时使用此技能"

第二种写法告诉 Claude：什么情况下你该被激活。Claude 每次收到用户消息，都会扫描所有 Skill 的 description，匹配最相关的那个。如果你写的是功能说明，Claude 不知道什么时候该用你。

7

记忆与数据存储

有些 Skills 可以通过存储数据实现记忆——最简单的方式是一个只追加写入的文本日志文件或 JSON 文件。

文章举了 standup-post 保存每次站会记录的例子。翻译到你的场景：你的日报 Skill 每天生成报告后，把关键指标追加写入一个 history.jsonl：

{"date":"2026-03-21","cups":3280,"orders":2100,"new_users":45,"revenue":12500}
{"date":"2026-03-22","cups":3150,"orders":1980,"new_users":38,"revenue":11800}
{"date":"2026-03-23","cups":3420,"orders":2200,"new_users":52,"revenue":13200}

明天 Claude 生成日报时，自动读取历史数据：

  • 自动算周同比（不用你手动给基准值）
  • 自动标注异常值（"今天新客数比7天均值高37%"）
  • 自动识别趋势（"连续3天杯量上升"）

这就是 Skill 的"记忆"——让 AI 不再是每次都失忆的工具，而是一个积累了工作经验的助手。

8

存储脚本与生成代码

给 Claude 提供脚本和库，让它把精力花在组合编排上——决定下一步做什么，而不是重新构造样板代码。

这是解决你"维度一直在变，SQL 没法固化"问题的关键思路。不固化最终 SQL，固化原子取数模块：

# modules/get_cups.py
def get_daily_cups(date, shop_filter=None):
    """从 ADS 表取门店杯量"""
    sql = f"""
    SELECT shop_name, SUM(sku_cnt) AS cups, SUM(order_cnt) AS orders
    FROM dw_ads.ads_mg_sku_shop_sales_statistic_d_1d
    WHERE dt = '{date}' AND tenant = 'LKUS'
      AND one_category_name = 'Drink'
      AND shop_name NOT IN ('NJ Test Kitchen', 'NJ Test Kitchen 2')
    """
    if shop_filter:
        sql += f" AND shop_name = '{shop_filter}'"
    sql += " GROUP BY shop_name ORDER BY cups DESC"
    return sql

# modules/get_traffic.py

你的维度会变，但积木块是稳定的。变化的部分交给 Claude 临场发挥，稳定的部分用 Skill 固化。

第四部分

分发与组合

怎么分享 Skills

文章提供两种方式：

  1. 提交到代码仓库（放在 .claude/skills/ 下）——适合小团队
  2. 做成插件市场——适合大规模分发

对你来说，第一种就够了。你现在的 Skills 都在 ~/.claude/skills/ 下，自己用。

但文章有一个重要提醒：每个提交进去的 Skill 都会给模型的上下文增加一点负担。如果你的 Skill 太多、太大，Claude 启动时上下文就被占满了，留给实际工作的空间就少了。

这就是为什么"渐进式披露"很重要——Skill 的主文件要精简，详细信息放子文件里按需加载。

Skills 怎么互相依赖

文章说：目前没有正式的依赖管理，但你可以直接按名字引用其他 Skills。

翻译到你的场景：你的 ab-experiment Skill 在做实验分析时，需要用到 luckyus-data-query 里的表结构信息。你不需要把表结构复制一份到 ab-experiment 里——直接在 Skill 指令里写"表结构参考 luckyus-data-query 的 knowledge-base.md"，Claude 会自动去读。

怎么衡量 Skills 效果

文章说他们用 PreToolUse 钩子记录 Skill 使用频率。

对你更实用的衡量方式：看你让 Claude 重做的次数。如果一个 Skill 上线后，某类分析的重做率从 3 次降到 1 次，说明 Skill 有效。

第五部分

按需钩子——一个容易忽略的高级功能

文章专门提了一个概念叫按需钩子（On Demand Hooks）——只在 Skill 被调用时才激活，整个会话期间保持生效。

两个例子：

  • /careful：拦截 rm -rf、DROP TABLE、force-push 等危险操作。你只在操作生产环境时才需要。
  • /freeze：阻止对特定目录之外的任何编辑。调试时用——"我想加日志但老是不小心改了不相关的代码"。

翻译到你的场景：你可以做一个 /production-mode 钩子——当你在跑生产数据时激活，它会：

  • 强制所有 SQL 都带 tenant = 'LKUS' 过滤
  • 禁止 DELETE/UPDATE/INSERT 操作（只允许 SELECT）
  • 自动加 LIMIT 1000 防止拉出超大结果集

平时关着不碍事，需要时说一声 /production-mode 就开启防护。

结语

Skill 就是你的经验文件夹化

Skill 就是把你脑子里那套"做某件事的全部经验"
——包括该怎么做、不该怎么做、做了会怎样——
文件夹化之后交给 AI。
Anthropic 内部几百个 Skills，本质上就是几百个工程师把各自的专业经验沉淀成了文件夹。

你不需要一次做完 14 个 Skills。文章最后说的那句话才是最重要的：

"我们大多数 Skills 一开始就是几行文字加一个踩坑点，后来因为大家不断补充 Claude 遇到的新边界情况，才慢慢变好的。"

先做一个。从你最常踩的那个坑开始。
