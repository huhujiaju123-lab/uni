## 整体框架

### 三层人群划分逻辑

```
第一层：价格策略
    ├─ 涨价组（30%）
    └─ 不涨价组（70%）
         ↓
第二层：生命周期（正交）
    ├─ 新客
    ├─ 0-15天老客
    ├─ 16-30天老客
    └─ 31天+老客
         ↓
第三层：触发策略
    ├─ 来访未购
    ├─ 上月消费本月未消费
    └─ 周六日消费周日发券
```

## 第一层：涨价/不涨价分流

### 分流规则

**目标：**
- 确保涨价组人群长期稳定固定（30%）
- 不涨价组作为对照组（70%）

**分流方法：**
```
方法1：用户ID哈希取模
- hash(user_id) % 100
- 0-29 → 涨价组（30%）
- 30-99 → 不涨价组（70%）

方法2：UUID随机分配并固化
- 首次进入实验时随机分配
- 分配结果写入离线表，永久固定
```

**关键要求：**
- ✓ 一次分配，永久固定
- ✓ 涨价组人群不随生命周期变化而窜组
- ✓ 分流算法稳定，重启实验不改变

### 两组对比

| 特征 | 涨价组（30%） | 不涨价组（70%） |
|------|--------------|----------------|
| 人群比例 | 30% | 70% |
| 价格策略 | 执行涨价 | 维持原价 |
| 稳定性 | 永久固定 | 永久固定 |
| 作用 | 实验组 | 对照组 |

## 第二层：生命周期正交

### 生命周期定义

**4个生命周期分组：**

1. **新客**
   - 定义：首次购买后0天
   - 特点：刚转化，敏感度未知

2. **0-15天老客**
   - 定义：首次购买后1-15天
   - 特点：新手期，培养忠诚度关键期

3. **16-30天老客**
   - 定义：首次购买后16-30天
   - 特点：过渡期，习惯养成期

4. **31天+老客**
   - 定义：首次购买后31天以上
   - 特点：成熟用户，价格敏感度稳定

### 正交矩阵

| 生命周期 | 涨价组（30%） | 不涨价组（70%） |
|---------|--------------|----------------|
| **新客** | 涨价_新客 | 原价_新客 |
| **0-15天** | 涨价_0-15 | 原价_0-15 |
| **16-30天** | 涨价_16-30 | 原价_16-30 |
| **31天+** | 涨价_31+ | 原价_31+ |

**总共形成 8 个基础人群组：**

```
涨价组（30%）：
  G1: 涨价_新客
  G2: 涨价_0-15天
  G3: 涨价_16-30天
  G4: 涨价_31天+

不涨价组（70%）：
  C1: 原价_新客
  C2: 原价_0-15天
  C3: 原价_16-30天
  C4: 原价_31天+
```

## 第三层：策略触发条件

### 三种触发策略

在上述8个基础组的基础上，叠加3种触发策略：

#### 策略1：来访未购

**触发条件：**
- 用户在过去7天内有访问行为
- 但未产生购买

**动作：**
- 发放专属优惠券
- 推送个性化推荐

**适用人群：**
- 所有8个基础组

---

#### 策略2：上月消费本月未消费

**触发条件：**
- 上个自然月有消费记录
- 本月至今无消费

**动作：**
- 召回优惠券
- 短信/push提醒

**适用人群：**
- 仅适用于老客（排除新客）
- 6个老客组：G2, G3, G4, C2, C3, C4

---

#### 策略3：周六日消费周日发券

**触发条件：**
- 用户在周六或周日产生消费

**动作：**
- 在下一个周日发放优惠券
- 鼓励周末复购

**适用人群：**
- 所有8个基础组

## 完整人群矩阵

### 矩阵维度

```
维度1：价格策略（2种）
  - 涨价组（30%）
  - 不涨价组（70%）

维度2：生命周期（4种）
  - 新客
  - 0-15天
  - 16-30天
  - 31天+

维度3：触发策略（3种）
  - 来访未购
  - 上月消费本月未消费
  - 周六日消费周日发券
```

### 人群总数

**基础人群组：** 2 × 4 = **8组**

**触发策略叠加后的细分：**
- 每个基础组可能触发0-3种策略
- 理论上最多：8 × 2^3 = 64种状态

**实际运营中的组合：**

| 基础组 | 来访未购 | 上月消费本月未 | 周六日发券 | 组合标签 |
|-------|---------|---------------|-----------|---------|
| 涨价_新客 | ✓ | - | ✓ | G1_来访_周末 |
| 涨价_新客 | ✗ | - | ✗ | G1_常规 |
| 涨价_0-15 | ✓ | ✓ | ✓ | G2_来访_召回_周末 |
| ... | ... | ... | ... | ... |

## 实施细节

### 阶段1：基础分流（1月30日前完成）

**步骤：**

1. **清理环境**
   ```
   时间：1月30日
   动作：
   - 下线所有现有价格实验
   - 下线冲突的优惠券活动
   - 清空AB实验配置
   ```

2. **圈选非新客人群**
   ```
   SQL示例：
   SELECT user_id
   FROM users
   WHERE first_purchase_date IS NOT NULL
   ```

3. **执行涨价/不涨价分流**
   ```
   方法：哈希取模

   涨价组（30%）：
   SELECT user_id
   FROM users
   WHERE MOD(ABS(HASH(user_id)), 100) < 30

   不涨价组（70%）：
   SELECT user_id
   FROM users
   WHERE MOD(ABS(HASH(user_id)), 100) >= 30
   ```

4. **导出离线人群包**
   ```
   文件：
   - price_increase_group_30pct.csv（涨价组）
   - price_control_group_70pct.csv（对照组）

   字段：
   - user_id
   - group（price_increase / price_control）
   - assigned_date（分配日期）
   ```

### 阶段2：生命周期分层（实时计算）

**计算逻辑：**

```sql
-- 每日更新用户生命周期标签
UPDATE user_segments
SET lifecycle_stage =
  CASE
    WHEN days_since_first_purchase = 0 THEN 'new_customer'
    WHEN days_since_first_purchase BETWEEN 1 AND 15 THEN 'days_0_15'
    WHEN days_since_first_purchase BETWEEN 16 AND 30 THEN 'days_16_30'
    WHEN days_since_first_purchase >= 31 THEN 'days_31_plus'
  END
WHERE user_id IN (涨价组 UNION 对照组)
```

**组合标签：**
```sql
-- 生成最终人群标签
SELECT
  user_id,
  CONCAT(price_group, '_', lifecycle_stage) AS segment
FROM user_segments
WHERE price_group IS NOT NULL

-- 输出示例：
-- 'price_increase_new_customer'
-- 'price_increase_days_0_15'
-- 'price_control_days_31_plus'
```

### 阶段3：触发策略配置（每日执行）

#### 策略1：来访未购

```sql
-- 每日识别来访未购用户
SELECT user_id, segment
FROM user_segments
WHERE last_visit_date >= DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY)
  AND last_purchase_date < DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY)

-- 添加标签
UPDATE user_segments
SET trigger_tags = CONCAT(trigger_tags, ',visit_no_purchase')
WHERE user_id IN (上述结果)
```

#### 策略2：上月消费本月未消费

```sql
-- 每月1日执行
SELECT user_id, segment
FROM user_segments
WHERE DATE_FORMAT(last_purchase_date, '%Y-%m') = DATE_FORMAT(DATE_SUB(CURRENT_DATE, INTERVAL 1 MONTH), '%Y-%m')
  AND DATE_FORMAT(last_purchase_date, '%Y-%m') < DATE_FORMAT(CURRENT_DATE, '%Y-%m')
  AND lifecycle_stage != 'new_customer'  -- 排除新客

-- 添加标签
UPDATE user_segments
SET trigger_tags = CONCAT(trigger_tags, ',monthly_lapse')
WHERE user_id IN (上述结果)
```

#### 策略3：周六日消费周日发券

```sql
-- 每周一执行（检查上周末消费）
SELECT user_id, segment
FROM orders
WHERE order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY)
  AND DAYOFWEEK(order_date) IN (1, 7)  -- 周日=1, 周六=7

-- 添加标签（在周日发券）
UPDATE user_segments
SET trigger_tags = CONCAT(trigger_tags, ',weekend_coupon')
WHERE user_id IN (上述结果)
  AND DAYOFWEEK(CURRENT_DATE) = 1  -- 仅在周日执行
```

**分流均衡性检查：**
```sql
-- 检查涨价/不涨价比例
SELECT
  price_group,
  COUNT(*) AS user_count,
  COUNT(*) / (SELECT COUNT(*) FROM user_segments) AS percentage
FROM user_segments
GROUP BY price_group

-- 期望结果：
-- price_increase: ~30%
```

## 风险控制

### 风险点与应对

**风险1：涨价导致大量流失**
- **监控阈值**：涨价组7日留存率 < 对照组 - 10%
- **应对措施**：
  - 立即暂停涨价
  - 对流失用户发放补偿券
  - 调整涨价幅度

**风险2：人群窜组**
- **监控**：定期检查用户是否仍在原分配组
- **应对**：技术锁定，禁止重新分配

**风险3：策略叠加过多**
- **监控**：同一用户同时触发3种策略
- **应对**：设置优先级，最多触发2种
