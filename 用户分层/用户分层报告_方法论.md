# 咖啡行业用户分层方法论框架

> **适用场景**：Lucky US（瑞幸美国）用户分析与精细化运营
> **数据基础**：Doris 数据库，租户标识 `tenant = 'LKUS'`
> **业务背景**：30+ 门店，主要覆盖纽约/新泽西地区

---

## 一、RFM 模型框架

### 1.1 核心指标定义

| 指标 | 英文全称 | 业务含义 | 计算逻辑 |
|------|---------|---------|---------|
| **R** | Recency | 最近一次消费距今天数 | `DATEDIFF(CURRENT_DATE, 最后订单日期)` |
| **F** | Frequency | 统计周期内的消费频次 | `COUNT(DISTINCT order_id)` |
| **M** | Monetary | 统计周期内的消费金额 | `SUM(pay_money)` |

### 1.2 阈值划分建议（基于咖啡行业特性）

#### R（最近消费天数）
```
高价值（R=3）: ≤7 天    # 1周内有消费，高活跃
中价值（R=2）: 8-30 天  # 1个月内有消费，中等活跃
低价值（R=1）: >30 天   # 超过1个月，沉睡/流失风险
```

**行业洞察**：咖啡属于高频消费品，理想客户每周消费2-3次。超过30天未消费，流失风险显著上升。

#### F（消费频次）
```
高频（F=3）: ≥8 次/月   # 周均2次以上，核心用户
中频（F=2）: 3-7 次/月  # 周均1次左右，普通用户
低频（F=1）: ≤2 次/月   # 偶尔尝试，边缘用户
```

**统计周期**：建议使用近30天或近90天。30天适合看短期活跃，90天适合稳定分层。

#### M（消费金额）
```
高消费（M=3）: ≥$150/月  # 月均$150+，约12杯+ ($12/杯)
中消费（M=2）: $50-149   # 月均$50-149，约4-11杯
低消费（M=1）: <$50      # 月均<$50，低贡献
```

**价格参考**：Lucky US 饮品客单价约 $10-15，按月均8-12杯消费作为高价值标准。

### 1.3 RFM 八大用户群体

基于 RFM 组合（3×3×3=27 种），合并为8大核心群体：

| 用户群体 | RFM 组合 | 特征描述 | 运营策略 | 占比预估 |
|---------|---------|---------|---------|---------|
| **🏆 重要价值客户** | R3-F3-M3 | 近期消费、高频高额 | **VIP权益**：专属折扣、新品试饮、会员日 | 5-10% |
| **🔥 重要发展客户** | R3-F2/3-M1/2 | 近期活跃但金额未达标 | **促进消费**：加价购、套餐优惠、提升客单价 | 15-20% |
| **💤 重要挽留客户** | R1-F2/3-M2/3 | 曾经高价值，但近期流失 | **召回激活**：专属券、流失预警、个性化推荐 | 8-12% |
| **⚡ 一般价值客户** | R2-F2-M2 | 中等活跃度和消费 | **保持维护**：常规促销、会员积分、节日营销 | 25-30% |
| **📈 一般发展客户** | R2/3-F1-M1 | 近期有消费但频次低 | **培养习惯**：新客福利、组合套餐、饮品推荐 | 15-20% |
| **🆕 新客户** | R3-F1-M1 | 首次或少数消费 | **快速转化**：新人券、首单优惠、引导第二单 | 10-15% |
| **💧 低价值客户** | R1-F1-M1 | 很久未消费，低频低额 | **低成本触达**：批量push、通用优惠券 | 10-15% |
| **❌ 流失客户** | R1（>90天） | 长期未消费 | **成本可控召回**：大力度券、问卷调研 | 5-10% |

### 1.4 SQL 实现示例

```sql
-- RFM 分层 SQL（基于 Lucky US 数据结构）
WITH user_rfm AS (
  SELECT
    user_id,
    -- R: 最近消费距今天数
    DATEDIFF(CURRENT_DATE, MAX(order_date)) AS recency,
    -- F: 近90天消费频次
    COUNT(DISTINCT order_id) AS frequency,
    -- M: 近90天消费金额
    SUM(pay_money) AS monetary
  FROM dw_ads.ads_user_order_info_d_nm
  WHERE dt = '${最新分区日期}'
    AND tenant = 'LKUS'
    AND status = 90  -- 成功订单
    AND order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  GROUP BY user_id
),
rfm_score AS (
  SELECT
    user_id,
    recency, frequency, monetary,
    -- R 打分（天数越小越好，所以逻辑相反）
    CASE
      WHEN recency <= 7 THEN 3
      WHEN recency <= 30 THEN 2
      ELSE 1
    END AS r_score,
    -- F 打分
    CASE
      WHEN frequency >= 8 THEN 3
      WHEN frequency >= 3 THEN 2
      ELSE 1
    END AS f_score,
    -- M 打分
    CASE
      WHEN monetary >= 150 THEN 3
      WHEN monetary >= 50 THEN 2
      ELSE 1
    END AS m_score
  FROM user_rfm
)
SELECT
  user_id,
  CONCAT(r_score, f_score, m_score) AS rfm_code,
  CASE
    WHEN r_score = 3 AND f_score = 3 AND m_score = 3 THEN '重要价值客户'
    WHEN r_score = 3 AND f_score >= 2 AND m_score <= 2 THEN '重要发展客户'
    WHEN r_score = 1 AND f_score >= 2 AND m_score >= 2 THEN '重要挽留客户'
    WHEN r_score = 2 AND f_score = 2 AND m_score = 2 THEN '一般价值客户'
    WHEN r_score >= 2 AND f_score = 1 AND m_score = 1 THEN '一般发展客户'
    WHEN r_score = 3 AND f_score = 1 AND m_score = 1 THEN '新客户'
    WHEN r_score = 1 AND f_score = 1 AND m_score = 1 THEN '低价值客户'
    WHEN r_score = 1 THEN '流失客户'
    ELSE '其他'
  END AS user_segment
FROM rfm_score;
```

## 二、用户生命周期模型

### 2.1 生命周期阶段定义

| 阶段 | 定义 | 关键特征 | 典型行为 | 停留时长 |
|------|------|---------|---------|---------|
| **🌱 新客期** | 注册后首次消费 | 首单完成，体验产品 | 首单优惠券使用，试探性消费 | 0-7 天 |
| **🚀 成长期** | 开始形成消费习惯 | 第2-5单，频次上升 | 尝试不同品类，优惠券依赖 | 7-30 天 |
| **💎 成熟期** | 稳定高价值贡献 | 6单以上，高频稳定 | 固定偏好，自然消费为主 | 30+ 天 |
| **😴 沉睡期** | 活跃度下降 | 30-60天未消费 | 偶尔打开APP，未下单 | 30-60 天 |
| **💔 流失期** | 基本停止消费 | 60天以上未消费 | 不再打开APP | 60+ 天 |

### 2.2 各阶段关键指标

#### 新客期（0-7天）
```
✓ 首单转化率：注册用户中完成首单的比例
✓ 首单客单价：反映产品吸引力
✓ 首单到二单天数：习惯养成速度
✓ 目标：首单转化率 >60%，7日内完成二单 >40%
```

#### 成长期（7-30天）
```
✓ 复购率：完成2单以上的用户占比
✓ 品类尝试度：尝试不同饮品的种类数
✓ 优惠券使用率：对促销的敏感度
✓ 目标：7日复购率 >30%，30日复购率 >50%
```

#### 成熟期（30+天）
```
✓ 月均消费频次：稳定在 4-8 次/月
✓ LTV（生命周期价值）：累计消费金额
✓ NPS（净推荐值）：推荐意愿
✓ 目标：月活用户中成熟用户占比 >40%
```

#### 沉睡期（30-60天未消费）
```
✓ 沉睡率：30天未消费用户占活跃用户比例
✓ 唤醒率：沉睡用户重新下单的比例
✓ 目标：沉睡率 <15%，唤醒率 >25%
```

#### 流失期（60+天未消费）
```
✓ 流失率：60天未消费用户占比
✓ 召回成本：召回一个流失用户的平均成本
✓ 召回ROI：召回用户带来的LTV vs 召回成本
✓ 目标：流失率 <10%，召回ROI >3:1
```

### 2.3 生命周期流转逻辑

```
[新客] → [成长期] → [成熟期] ⇄ [沉睡期] → [流失期]
  ↓         ↓          ↓          ↓          ↓
15%流失   20%流失   5%流失    40%流失   80%不回
85%进入   80%进入   95%保持   60%唤醒   20%召回
```

### 2.4 SQL 实现示例

```sql
-- 用户生命周期分层 SQL
WITH user_order_agg AS (
  SELECT
    user_id,
    MIN(order_date) AS first_order_date,
    MAX(order_date) AS last_order_date,
    COUNT(DISTINCT order_id) AS total_orders,
    DATEDIFF(CURRENT_DATE, MAX(order_date)) AS days_since_last_order
  FROM dw_ads.ads_user_order_info_d_nm
  WHERE dt = '${最新分区日期}'
    AND tenant = 'LKUS'
    AND status = 90
  GROUP BY user_id
)
SELECT
  user_id,
  CASE
    WHEN total_orders = 1 AND days_since_last_order <= 7 THEN '新客期'
    WHEN total_orders <= 5 AND days_since_last_order <= 30 THEN '成长期'
    WHEN total_orders >= 6 AND days_since_last_order <= 30 THEN '成熟期'
    WHEN days_since_last_order BETWEEN 30 AND 60 THEN '沉睡期'
    WHEN days_since_last_order > 60 THEN '流失期'
    ELSE '其他'
  END AS lifecycle_stage
FROM user_order_agg;
```

## 三、价值金字塔模型

### 3.1 三层金字塔结构

```
        /\  头部高价值（Top 10%）
       /  \  ↑ 贡献 60-70% GMV
      /────\  中部活跃（Middle 30%）
     / ────ㅡ\  ↑ 贡献 25-30% GMV
    /────────\  底部低频（Bottom 60%）
   /──────────\  ↑ 贡献 5-10% GMV
```

### 3.2 分层定义

| 层级 | 用户占比 | GMV贡献 | 特征描述 | 月均消费 |
|------|---------|---------|---------|---------|
| **👑 头部高价值** | 10% | 60-70% | 高频高额，品牌忠诚 | $200+ |
| **🔶 中部活跃** | 30% | 25-30% | 中频中额，稳定消费 | $50-199 |
| **🔹 底部低频** | 60% | 5-10% | 低频低额，偶尔尝试 | <$50 |

### 3.3 运营策略差异化

#### 头部高价值（10%）
```
策略：精细化服务 + 情感连接
- 🎁 VIP专属权益：生日福利、新品试饮、会员日折扣
- 📞 1对1服务：门店专属客服、定制推荐
- 💌 情感营销：感谢信、年度消费报告、社群运营
- 🎯 目标：提升NPS，增强粘性，延长LTV
```

#### 中部活跃（30%）
```
策略：标准化运营 + 向上转化
- 🎟️ 常规促销：满减券、加价购、套餐优惠
- 📊 行为引导：推荐高客单价产品、鼓励多品类尝试
- 🏆 积分体系：消费积分、等级升级、兑换权益
- 🎯 目标：提升客单价和频次，向头部迁移
```

#### 底部低频（60%）
```
策略：低成本触达 + 批量激活
- 📱 自动化营销：批量push、通用优惠券
- 🔄 召回策略：流失预警、大力度折扣
- 💰 成本控制：ROI为导向，避免过度投入
- 🎯 目标：激活其中15-20%向中部迁移
```

### 3.4 SQL 实现示例

```sql
-- 价值金字塔分层 SQL（按90天消费金额）
WITH user_value AS (
  SELECT
    user_id,
    SUM(pay_money) AS total_spend_90d,
    NTILE(10) OVER (ORDER BY SUM(pay_money) DESC) AS decile_rank
  FROM dw_ads.ads_user_order_info_d_nm
  WHERE dt = '${最新分区日期}'
    AND tenant = 'LKUS'
    AND status = 90
    AND order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  GROUP BY user_id
)
SELECT
  user_id,
  total_spend_90d,
  CASE
    WHEN decile_rank = 1 THEN '头部高价值（Top 10%）'
    WHEN decile_rank <= 4 THEN '中部活跃（Middle 30%）'
    ELSE '底部低频（Bottom 60%）'
  END AS value_tier
FROM user_value;
```

## 四、咖啡行业特有维度

### 4.1 品类偏好分层

基于用户主力消费品类的偏好分类：

| 品类类型 | 定义 | 用户占比 | 客单价 | 运营策略 |
|---------|------|---------|--------|---------|
| **☕ 经典美式党** | 60%+ 订单为美式/冷萃 | 20-25% | $8-10 | 简单高效，强调品质和性价比 |
| **🥛 奶咖爱好者** | 60%+ 订单为拿铁/卡布 | 35-40% | $11-13 | 口感丝滑，推荐燕麦奶/厚乳 |
| **🎨 特调尝鲜派** | 爱点季节限定/创意饮品 | 15-20% | $13-16 | 新品驱动，强调颜值和话题性 |
| **🍵 非咖啡族** | 主要点茶饮/果汁 | 10-15% | $9-12 | 扩展品类，满足多元需求 |
| **🔀 百变选手** | 各品类均衡消费 | 10-15% | $12-14 | 推荐组合套餐，提升尝试度 |

**SQL 示例**：
```sql
-- 品类偏好分层（基于 SKU 大类统计）
WITH user_category AS (
  SELECT
    user_id,
    one_category_name,
    COUNT(DISTINCT order_id) AS orders_per_category,
    SUM(COUNT(DISTINCT order_id)) OVER (PARTITION BY user_id) AS total_orders
  FROM dw_ads.ads_user_order_info_d_nm
  WHERE dt = '${最新分区日期}'
    AND tenant = 'LKUS'
    AND status = 90
    AND order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  GROUP BY user_id, one_category_name
),
user_preference AS (
  SELECT
    user_id,
    one_category_name AS primary_category,
    orders_per_category * 100.0 / total_orders AS category_ratio,
    ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY orders_per_category DESC) AS rn
  FROM user_category
)
SELECT
  user_id,
  primary_category,
  category_ratio,
  CASE
    WHEN primary_category IN ('美式', 'Americano', '冷萃') AND category_ratio >= 60 THEN '经典美式党'
    WHEN primary_category IN ('拿铁', 'Latte', '卡布奇诺') AND category_ratio >= 60 THEN '奶咖爱好者'
    WHEN primary_category LIKE '%限定%' OR primary_category LIKE '%特调%' THEN '特调尝鲜派'
    WHEN primary_category IN ('茶饮', '果汁', 'Tea', 'Juice') THEN '非咖啡族'
    ELSE '百变选手'
  END AS preference_segment
FROM user_preference
WHERE rn = 1;
```

### 4.2 价格敏感度分层

基于优惠券使用行为的价格敏感度分类：

| 敏感度 | 定义 | 用户占比 | 特征 | 运营策略 |
|--------|------|---------|------|---------|
| **💎 价格无感** | 券使用率 <20% | 10-15% | 自然消费，品牌忠诚 | 减少促销打扰，强化品牌价值 |
| **⚖️ 理性消费** | 券使用率 20-60% | 50-60% | 有券用券，无券也买 | 常规促销即可，重点维护满意度 |
| **🎟️ 优惠依赖** | 券使用率 >60% | 25-30% | 几乎仅在有券时消费 | 控制券成本，引导自然消费 |

**SQL 示例**：
```sql
-- 价格敏感度分层（基于优惠券使用率）
WITH user_coupon_usage AS (
  SELECT
    user_id,
    COUNT(DISTINCT order_id) AS total_orders,
    COUNT(DISTINCT CASE WHEN discount_amount > 0 THEN order_id END) AS coupon_orders,
    COUNT(DISTINCT CASE WHEN discount_amount > 0 THEN order_id END) * 100.0
      / COUNT(DISTINCT order_id) AS coupon_usage_rate
  FROM dw_ads.ads_user_order_info_d_nm
  WHERE dt = '${最新分区日期}'
    AND tenant = 'LKUS'
    AND status = 90
    AND order_date >= DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  GROUP BY user_id
)
SELECT
  user_id,
  coupon_usage_rate,
  CASE
    WHEN coupon_usage_rate < 20 THEN '价格无感'
    WHEN coupon_usage_rate < 60 THEN '理性消费'
    ELSE '优惠依赖'
  END AS price_sensitivity
FROM user_coupon_usage;
```

### 4.3 消费时段偏好分层

基于下单时段的场景分类：

| 时段类型 | 定义 | 用户占比 | 场景描述 | 运营策略 |
|---------|------|---------|---------|---------|
| **🌅 早餐刚需** | 60%+ 订单在 7-10am | 30-35% | 通勤路上/到公司 | 早餐套餐、快速出餐 |
| **☀️ 午后续命** | 60%+ 订单在 2-4pm | 25-30% | 下午茶提神 | 冰饮特调、甜品组合 |
| **🌙 全天候党** | 订单分布均匀 | 20-25% | 无固定场景 | 全时段运营，提升曝光 |
| **🌃 夜猫偏好** | 30%+ 订单在 8pm后 | 10-15% | 加班/深夜学习 | 低咖啡因饮品、深夜专属 |

**SQL 示例**：
```sql
-- 消费时段偏好分层（基于下单时段分布）
WITH user_hour_dist AS (
  SELECT
    user_id,
    HOUR(CONVERT_TZ(create_time, @@time_zone, 'America/New_York')) AS order_hour,
    COUNT(DISTINCT order_id) AS orders_per_hour,
    SUM(COUNT(DISTINCT order_id)) OVER (PARTITION BY user_id) AS total_orders
  FROM dw_ods.t_order
  WHERE tenant = 'LKUS'
    AND status = 90
    AND DATE(CONVERT_TZ(create_time, @@time_zone, 'America/New_York'))
        >= DATE_SUB(CURRENT_DATE, INTERVAL 90 DAY)
  GROUP BY user_id, order_hour
),
user_time_preference AS (
  SELECT
    user_id,
    SUM(CASE WHEN order_hour BETWEEN 7 AND 10 THEN orders_per_hour ELSE 0 END) * 100.0 / total_orders AS morning_ratio,
    SUM(CASE WHEN order_hour BETWEEN 14 AND 16 THEN orders_per_hour ELSE 0 END) * 100.0 / total_orders AS afternoon_ratio,
    SUM(CASE WHEN order_hour >= 20 THEN orders_per_hour ELSE 0 END) * 100.0 / total_orders AS night_ratio
  FROM user_hour_dist
  GROUP BY user_id, total_orders
)
SELECT
  user_id,
  CASE
    WHEN morning_ratio >= 60 THEN '早餐刚需'
    WHEN afternoon_ratio >= 60 THEN '午后续命'
    WHEN night_ratio >= 30 THEN '夜猫偏好'
    ELSE '全天候党'
  END AS time_preference
FROM user_time_preference;
```

## 五、综合应用：多维度分层矩阵

### 5.1 分层组合策略

在实际运营中，建议组合多个维度形成立体的用户画像：

| 用户群体示例 | RFM | 生命周期 | 价值层级 | 品类偏好 | 价格敏感 | 运营优先级 |
|------------|-----|---------|---------|---------|---------|-----------|
| **核心VIP** | 333 | 成熟期 | 头部10% | 百变选手 | 价格无感 | ⭐⭐⭐⭐⭐ |
| **潜力新星** | 321 | 成长期 | 中部30% | 特调尝鲜 | 理性消费 | ⭐⭐⭐⭐ |
| **沉睡大户** | 133 | 沉睡期 | 头部10% | 奶咖爱好 | 优惠依赖 | ⭐⭐⭐⭐ |
| **普通用户** | 222 | 成熟期 | 中部30% | 美式党 | 理性消费 | ⭐⭐⭐ |
| **低价值观望** | 111 | 新客期 | 底部60% | 非咖啡族 | 优惠依赖 | ⭐ |

### 5.2 分层决策树

```
用户分层决策逻辑：
1. 先看生命周期阶段 → 确定用户状态（新客/活跃/流失）
2. 再看 RFM 得分 → 确定价值潜力（高/中/低）
3. 结合品类偏好 → 精准推荐产品
4. 结合价格敏感度 → 设定促销策略
5. 结合时段偏好 → 优化触达时机
```

## 七、附录：方法论应用示例

### 示例 1：识别沉睡高价值用户进行召回

```sql
-- 目标：找出曾经高价值但近期沉睡的用户，进行定向召回
SELECT
  u.user_id,
  u.user_phone,
  rfm.recency AS 距上次消费天数,
  rfm.frequency AS 历史消费频次,
  rfm.monetary AS 历史消费金额,
  pref.primary_category AS 偏好品类,
  sens.coupon_usage_rate AS 优惠券使用率
FROM user_rfm rfm
LEFT JOIN user_preference pref ON rfm.user_id = pref.user_id
LEFT JOIN user_coupon_usage sens ON rfm.user_id = sens.user_id
LEFT JOIN dw_dim.dim_user u ON rfm.user_id = u.user_id
WHERE rfm.r_score = 1  -- 30天以上未消费
  AND rfm.f_score >= 2  -- 历史频次不低
  AND rfm.m_score >= 2  -- 历史金额不低
ORDER BY rfm.monetary DESC
LIMIT 1000;
```

**运营动作**：
- 定向推送专属召回券（如 $5 off $15）
- 短信文案："好久不见！您喜欢的{偏好品类}有新品啦，专属券已到账"
- 追踪召回率和召回ROI

### 示例 2：新客快速转化

```sql
-- 目标：找出完成首单但未完成二单的新客，推动复购
SELECT
  u.user_id,
  u.user_phone,
  first_order.order_date AS 首单日期,
  DATEDIFF(CURRENT_DATE, first_order.order_date) AS 距首单天数,
  first_order.sku_name AS 首单产品
FROM dw_ads.ads_user_order_info_d_nm agg
JOIN (
  SELECT user_id, order_date, sku_name
  FROM dw_ods.t_order_item oi
  JOIN dw_ods.t_order o ON oi.order_id = o.order_id
  WHERE o.tenant = 'LKUS' AND o.status = 90
) first_order ON agg.user_id = first_order.user_id
WHERE agg.dt = '${最新分区日期}'
  AND agg.total_orders = 1  -- 只有1单
  AND DATEDIFF(CURRENT_DATE, first_order.order_date) BETWEEN 3 AND 7  -- 首单后3-7天
LIMIT 5000;
```

**运营动作**：
- App push："首单体验满意吗？二单立减$3"
- 推荐与首单相关的品类："喜欢{首单产品}的用户也爱这款"
- 设置 7日内完成二单目标，追踪转化率

## 八、总结与建议

### 8.1 核心要点

1. **RFM 是基础**：快速识别高价值用户和流失风险，适合快速上手
2. **生命周期是动态视角**：关注用户成长路径和流转，指导长期运营
3. **价值金字塔是资源分配**：差异化投入，避免"撒胡椒面"
4. **行业特性是加分项**：品类/价格/时段偏好，让分层更精准

### 8.2 实施路线图

```
Week 1-2:  搭建 RFM 基础分层，输出 8 大用户群体
Week 3-4:  叠加生命周期和价值金字塔，形成立体画像
Week 5-6:  设计差异化运营方案，启动 A/B 测试
Week 7-8:  追踪效果数据，迭代优化阈值和策略
```

### 8.3 关键成功因素

✓ **数据质量**：确保订单/用户表数据准确，定期更新
✓ **阈值校准**：根据实际业务数据调整 RFM 阈值，避免生搬硬套
✓ **跨部门协同**：分层结果需与产品/市场/CRM 团队打通
✓ **持续迭代**：用户分层不是一次性项目，需每月复盘优化
