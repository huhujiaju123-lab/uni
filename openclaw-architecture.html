递归大师 — 技术改造蓝图

OpenClaw 本身是一个强大的 AI 编程助手。我们做的事情，是在它的标准能力之上加一层「自我进化」的机制——让它能从错误中学习、从重复中提炼经验、在团队间共享知识，越用越聪明。

为什么叫「递归大师」？因为整个系统的核心就是一个不断自我调用的循环：使用中犯错，犯错后记录，记录变成经验，经验指导下一次使用。每一轮都比上一轮更强，这就是递归。
使用 → 犯错 → 记录 → 学习 → 使用得更好 → ♻ 循环

SECTION 02 六大改造模块详解

每个模块解决一个具体的问题。下面逐一说明它「做什么」和「为什么需要」，技术细节折叠在备注里。

Module 1 — 错题本前置检查

在 AI 看到你的问题之前，先翻一遍自己的错题本，看看这个问题有没有已知的坑。

为什么需要？因为 AI 没有记忆——上次犯过的错，下次照犯不误。这个模块就是给它一个「考试前看错题集」的能力，把历史教训提前注入，避免重蹈覆辙。同时它还会检测你的问题是否和之前做过的某类任务很像，如果是，就推荐直接复用现有的技能模板，省得重新来一遍。

工作流程

流程很直观：你输入问题 → 脚本在后台快速扫描错误库和模式库 → 如果命中已知坑，就在问题旁边加一条提醒（比如「这个查询上次因为时区问题出过错，注意用 CONVERT_TZ」）→ 然后才把加了提醒的内容交给 AI 去思考。

你的问题 扫描错题本和模式库 注入提醒信息 AI 开始思考

Module 2 — 专家人格植入

把 AI 从一个「什么都能聊一点」的通用助手，变成一个「深度理解运营分析」的领域专家。

为什么需要？OpenClaw 默认是一个全能型助手，对所有领域一视同仁。但我们的工作场景非常具体——运营分析、数据查询、AB实验。通过一个叫 SOUL.md 的配置文件，我们把运营专家的思维方式、决策框架、领域知识都写进去，让 AI 每次启动时就「装载」这些专业能力。好比给一个聪明但没有行业经验的新人，发了一本详细的「运营分析手册」。

植入了什么

四个关键模块：身份定义（你是资深运营分析专家）、决策框架（遇到模糊需求先追问再行动、多方案时列 pros/cons）、推理协议（复杂问题必须分步推理、每个结论必须引用数据来源）、领域知识（杯量/客单价/复购率这些指标体系、环比/同比/漏斗这些分析方法论）。

技术实现细节：SOUL.md 内容预览
## Identity
你是一个资深运营分析专家，擅长用数据驱动决策。
## Decision Framework
面对模糊需求 → 先追问 → 列假设 → 确认后行动
面对多个方案 → 列 pros/cons → 可逆性优先
面对不确定信息 → 标注置信度（高/中/低）→ 不编造
## Reasoning Protocol
复杂问题必须分步推理
每个结论必须引用数据来源
发现矛盾时立即停下说明
## Domain Knowledge
运营指标体系：杯量 / 客单价 / 复购率 / 转化率
分析方法论：环比 / 同比 / 漏斗 / 归因
常见陷阱：口径不统一 / 幸存者偏差 / 辛普森悖论

Module 3 — 结构化学习体系

从「散装记忆」到「分层管理的知识体系」，让经验不仅能记住，还能沉淀和传播。

为什么需要？OpenClaw 本身已经有记忆能力（L1-L3），能在对话中记住东西、写日志、保留长期记忆。但这些记忆是「平铺」的，没有筛选和晋升机制。我们新增的 L4-L5 做了两件事：一是让重要经验能通过 Git 在团队间共享（L4 团队记忆），二是让反复验证过的核心经验直接写入 AI 的「性格」里（L5 DNA 记忆），成为它思考问题的本能。

五层记忆体系

可以把这五层想象成一个知识的「蒸馏塔」：最底层是海量的日常对话，往上逐渐浓缩，最顶层只留下最核心的认知。越往上越精炼、越持久、影响范围越大。

Module 4 — 负反馈引擎

每次犯错都是一次学习机会。这个模块确保错误不会白犯——自动记录、自动分类、自动升级。

为什么需要？AI 天生没有「痛感」。它犯了错不会难受，下次还会犯同样的错。负反馈引擎就是给它装了一个「痛觉神经」：出错了自动记下来，被用户纠正了也记下来，然后通过一套晋升规则，让频繁出现的错误从「临时便签」逐步升级为「铁律」，最终写入 AI 的 DNA。

什么时候会触发记录

四种场景会自动触发：

命令执行失败
AI 跑了一条命令结果报错，自动记录错误原因
用户纠正
你说了「不对」「别这样做」，系统捕捉到纠正语气并记录
每日自动回顾
每天结束时自动聚类分析，找出重复出现的错误模式
用户主动说「记住」
你主动告诉 AI 一条规则，直接写入对应的文件

不是所有错误都值得永久记住。这套机制用「出现频率」来决定一条经验的重要性，频率越高，保存层级越高：

Module 5 — 技能自进化引擎

如果你让 AI 做了三次类似的事情，它会主动把这类工作提炼成一个可复用的「技能模板」，以后一键调用。

为什么需要？日常工作中有大量重复劳动：每周跑类似的 SQL、每次做 AB 实验分析步骤都差不多、日报格式基本固定。与其每次从头教 AI，不如让它自己发现这些重复模式，主动提炼成技能。好比一个勤快的助理，干了三次同样的活之后，自己写了个 SOP 拿来给你审批。

工作流程

整个流程分五步：做事（正常执行任务）→ 计数（后台默默记录相似任务的出现次数）→ 触发（同一类任务出现 3 次以上）→ 生成草案（自动把步骤、模板、规则提炼成一个技能文档）→ 人工审批（你确认后才正式入库，不会自作主张）。

Module 6 — 团队知识同步

一个人踩过的坑、学到的经验、提炼的技能，自动同步给团队所有人，避免重复交学费。

为什么需要？如果每个人都在自己的机器上单独使用 AI 助手，那 A 踩过的坑 B 还会再踩一次。通过 Git 版本管理，我们把知识分成两类：共享区（人格设定、操作规则、技能库、错误库、团队记忆）所有人同步，一人学会全员受益；私有区（个人工作笔记）只在本地，不影响别人。

同步架构

三个团队成员各自有一份本地工作区，通过 Git 远程仓库保持同步。共享文件推送上去后，其他人 pull 就能获取最新的知识。私有文件通过 .gitignore 排除，完全不会互相干扰。

怎么处理冲突

不同类型的文件用不同的策略，确保不会「你改了我也改了然后冲突一团糟」：

错误库 (.learnings/)

只追加不修改。每条记录有唯一编号，两个人同时加记录也不会冲突。

人格设定 (SOUL.md)

必须通过 PR 审批。人格改动影响全团队，需要大家达成共识。

技能库 (skills/)

每个技能是独立目录。新建技能不会冲突，修改已有技能需走 PR。
